## 背景與現狀

本專案旨在建立一個名為「Engineer News」的技術決策分享平台，模仿 `quidproquo.cc` 的功能，並加入獨有的「對話即文件」工作流。基礎架構將利用 Astro 與 Cloudflare 的邊緣生態系統。

## 目標與非目標

**目標：**
- 使用 Astro 交付高性能、極簡主義的技術資訊流。
- 實施 AI 優先的內容策略 (`llms.txt`, JSON-LD, 結構化元數據)。
- 支持雙語內容 (繁中/英文) 並實現無縫切換。
- 建立 CLI 管道，將工程對話（如 Claude Code 日誌）轉換為結構化文章。
- 部署至 Cloudflare Pages，並利用 D1 處理輕量級動態功能。

**非目標：**
- 建立功能完整的瀏覽器後台 CMS（偏好基於 Git 的 Markdown 工作流）。
- 支援多人實時協作編輯。
- 實作社交功能如評論或點讚（MVP 階段不包含）。

## 技術決策

### 1. 框架：Astro
**理由**：Astro 針對以內容為主的網站進行了優化，客戶端 JavaScript 極少。其群島架構 (Island Architecture) 允許僅在需要的地方加入交互元素。
**備選方案**：Next.js（對部落格來說過於臃腫）、Hugo（對現代 Web 組件的支持較弱）。

### 2. 基礎設施：Cloudflare Pages + D1 + Vectorize
**理由**：Pages 提供零配置部署與邊緣性能。D1 (邊緣 SQLite) 非常適合搜尋索引與元數據儲存。Vectorize 則為未來的 RAG 功能提供基礎。
**備選方案**：Vercel（邊緣數據庫成本較高）、Netlify。

### 3. 內容管道：`opsx-ingest` CLI
**理由**：為了實現「對話即文件」，需要一個本地工具來解析對話日誌、清理敏感資訊，並利用 LLM 生成敘述性的技術文章結構。
**備選方案**：手動複製貼上（摩擦力太大）。

### 4. AI 優化：`llms.txt` 與 Workers AI
**理由**：確保網站對 AI 代理人友好。使用 Cloudflare 內建的 Workers AI 處理 Embedding，實現低延遲且低成本的語義搜尋。

## 風險與權衡

- **[風險]** 對話日誌包含敏感數據 → **[緩解]** 攝取工具必須包含「脫敏」步驟，在送交 LLM 處理前移除所有 Secret/Key。
- **[風險]** 雙語內容的維護成本 → **[緩解]** 攝取工具將支援自動化的翻譯建議流程。
- **[風險]** D1 的冷啟動或延遲 → **[緩解]** 保持大部分內容靜態化，僅在搜尋或數據統計等非關鍵路徑使用 D1。
